Bugg #1: Tillståndsbugg vid borttagning av läkemedelskort (medCardCount > 1)
I clearCurrentCard() (app.js + ui-renew.js): när man tar bort ett kort i mitten hanteras states.splice() och prescribeState omindexering, men calcDebounced[]-arrayen byggs om med gamla closures. Detta kan leda till att calc(i) anropas på fel index vid snabb växling.
Klinisk risk: Låg–medelhög (fel läkemedel visas i resultat).
Reproduktion: Lägg till 3 kort → ta bort kort nr 2 → växla till kort 3 → debounce kan trigga fel beräkning.

Race condition med pending debounce-timers efter borttagning av kort (utökning av din #1)
Plats: app.js: clearCurrentCard() + calcDebounced (array med closures).
Beskrivning:
Även om du gör calcDebounced.length = 0; + ensureDebounce-loop efter splice, kan redan queued debounce-timers (från input-event precis innan borttagning) fortfarande trigga med gamla closure-index. De muterar fel states[j] eller kör calc() på fel kort.
Reproduktion:
Lägg till 3 kort → skriv i kort #2 (debounce startar) → ta bort kort #2 snabbt → pending timer kör på fel index.
Klinisk risk: Låg–medel (fel läkemedel beräknas visas).
Fix-förslag: Byt till objekt-baserad debounce (debouncers = {} keyed by index) + clearTimeout innan splice.

Bugg #2: Vid viss omrendering (t.ex. switch mellan kort + debounce-cykel) kan flagsChanged triggas felaktigt och beslutet rensas. Läkaren måste då välja om:

Huvudlogiken ligger i calc-renew.js → funktionen calcCore(inputData, prev) (raderna med flagsChanged):JavaScriptconst flagsChanged = prev.isOveruse !== isOveruse || prev.isTooEarly !== isTooEarly;
const earlyRenewalDecision = flagsChanged ? null : (prev.earlyRenewalDecision || null);
Detta returneras och skrivs sedan över i calc() (samma fil):JavaScriptconst prev = {
  isOveruse:            s.isOveruse            || false,
  isTooEarly:           s.isTooEarly           || false,
  earlyRenewalDecision: s.earlyRenewalDecision || null,
};
const derived = calcCore(inputData, prev);
Object.assign(states[i], derived);   // ← här skrivs decision om till null

Exakt call-stack som triggar buggen:

Läkaren har valt "Ja, förnya" → setEarlyDecision('yes') (app.js) → sätter states[i].earlyRenewalDecision = 'yes'.
Någon omrendering sker:
selectMed(i) (ui-renew.js) → buildMedList() → renderFormForMed() → renderResultForMed()
ELLER debounce-cykeln i formuläret (formPanel.addEventListener('input') i app.js) → calcDebounced[activeMedIdx]() → calc()

calc() kör saveFormValues() (även om ingenting ändrats) → validateInputs() → calcCore().
I calcCore() jämförs prev.isOveruse/isTooEarly (från gamla state) med de nya beräknade värdena.
Om det finns minsta lilla skillnad (t.ex. getToday()-cache, mikro-skillnad i daysSince, parseDateUTC, eller att calculable flippar) → flagsChanged = true → earlyRenewalDecision = null.

Bugg #3: renderResultForMed antar ibland att s.metrics alltid finns (även om valid:true men calculable:false).
renderResultForMed antar ibland att s.metrics alltid finns (även när valid:true men calculable:false)
Var buggen sitter:

I ui-renew.js → renderResultForMed(i):JavaScriptif (!s.valid) { ... return; }   // ← här går vi förbi

/* Mätvärden */
const metricsGrid = getEl('metricsGrid');
if (metricsGrid && s.metrics) {   // ← skydd finns, men...
  metricsGrid.textContent = '';
  s.metrics.forEach(...)        // ← här antas att det är en array
}
Problemet uppstår när valid: true men calculable: false (t.ex. daysSince === 0).

Vad calcCore returnerar i det fallet:
JavaScriptreturn {
  valid: true,
  calculable: false,
  isOveruse: false,
  isTooEarly: false,
  metrics: [],          // ← explicit tom array
  alerts: [],
  ...
};
Varför det ändå kan gå fel:

Object.assign(states[i], derived) skriver inte alltid över metrics om det redan fanns en gammal värde i state.
I vissa tidiga tillstånd (t.ex. precis efter clearCurrentCard() eller innan första beräkningen är klar) kan s.metrics vara undefined.
if (metricsGrid && s.metrics) skyddar mot undefined, men om s.metrics är null eller en icke-array kan forEach krascha (eller skapa tom grid utan felmeddelande).
När calculable: false visas fortfarande hela resultContent (inklusive metricsGrid) – läkaren ser en tom eller felaktig mätvärdes-ruta.

Stale/ghost-data i UI efter ofullständiga return-värden från calcCore (hög risk, relaterad till men bredare än din #3)
Plats: calc-renew.js: calcCore() (rader ~140–220, alla early-return-paths) + app.js: calc() (Object.assign(states[i], derived)).
Beskrivning:
Många felvägar returnerar ofullständiga objekt:

daysSince === 0 → {valid:true, calculable:false, metrics:[] ...}
remaining > total / totalDays > 3650 / too_many_refs → ingen metrics, alerts, tlPct etc.
valid:false (incomplete/invalid_date) → nästan inget.

Object.assign behåller då gamla värden från tidigare giltig beräkning.
renderResultForMed() går förbi if(!s.valid) och renderar gamla metrics/alerts/tidslinje bredvid det nya verdictet ("Kan ej beräknas" / "Orimliga värden").

Bug #9: Felaktig månadsberäkning i Ny förskrivning (hög klinisk risk)
Plats: prescribe.js → calcPrescribeResult() (rader ~80–100, tYear / tMonth från today).
Beskrivning:
När nuvarande recept har dagar kvar (daysAlreadyCovered > 0) ska ny förskrivning starta från prescribedEndDate, inte från idag. Koden använder alltid today.getUTCFullYear() + today.getUTCMonth() vid "Månader"-läge.
Resultat: läkaren får för kort period (t.ex. 1 månad blir 1 månad minus redan täckta dagar).
Reproduktion:

Recept som räcker 20 dagar till.
Välj "Ny förskrivning" → 3 månader.
Beräkningen startar från idag → fel slutdatum och för få förpackningar.

Klinisk risk: Hög (fel förskrivningslängd → över-/underförskrivning).

Förbättring #7: Överdriven defensivitet vs. "Tyst misslyckande"
Observation: I JavaScript är det vanligt med if (s && s.valid). Koden har en bra balans, men kom ihåg regeln "föredra tydlig struktur framför överkommen defensivitet".
Svaghet: "Tysta misslyckanden" (early returns som inte uppdaterar UI) är en stor risk i kliniska verktyg. Om tillståndet är ogiltigt (s.valid === false), måste UI uppdateras så att felmeddelanden rensas eller en grå/låst ruta visas. Returnerar vi bara return; stannar applikationen kvar i sitt förra tillstånd, och läkaren kanske tror att det gamla resultatet gäller för den nya indatan.
